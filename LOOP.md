# Browserbase live demo loop

## Goal

Ship a Browserbase hosted-demo layer that matches the existing Cloudflare and
Google proof pattern: a reviewer can start a fresh Browserbase MCP proof run,
see verifier receipts, and inspect explorer plus log-proof links without
getting private Browserbase session material.

## Loop shape

Milestone persistence loop. This is a bounded checklist with receipts, not an
open-ended work session.

Maximum iterations per packet: 2. Stop if the same failure repeats twice.

## Packets

### Packet 1: contract and safety checkpoint

Scope:

- Confirm the worktree state and prior proof packet commit.
- Record the loop contract and hard stop rules.

Allowed files or surfaces:

- `LOOP.md`
- read-only inspection of existing Cloudflare, Google, Browserbase, and
  Firecrawl example files.

Expected artifact or diff:

- `LOOP.md` with this contract and receipt log.

Objective checks:

- `git status --branch --short`
- `git log -1 --oneline`

Human gates:

- None.

### Packet 2: Browserbase live demo implementation

Scope:

- Add a resettable Browserbase demo service that calls the existing
  Browserbase Stagehand proof runner path.
- Keep the proof packet and demo code separate.
- Expose a proof-first UI with run status, verifier result, record hashes,
  public log indexes, explorer links, and log-proof links.

Allowed files or surfaces:

- `packages/integration/examples/browserbase-stagehand/**`
- `packages/integration/package.json`
- `packages/integration/README.md`
- `packages/integration/demo-record-surfaces.json`
- `packages/integration/test/**`
- `proof-packets/browserbase-stagehand/README.md`
- `proof-packets/CONTACT_DRAFTS.md`
- `LOOP.md`

Expected artifact or diff:

- Browserbase demo server or Worker-style app.
- Browserbase demo README.
- Script to run the demo locally.
- UI or API route that creates a fresh bounded proof run.

Objective checks:

- Focused TypeScript compile for new files.
- Focused Vitest or API smoke test.
- Redaction scan for Browserbase keys, project id, replay URLs, selectors,
  page snapshots, and action instructions.

Human gates:

- Deploying the demo to a hosted URL.
- Creating or rotating external secrets.

### Packet 3: docs and contact copy

Scope:

- Update the Browserbase contact draft to include the live demo slot.
- Keep Firecrawl as ready for a GitHub comment, not blocked on a hosted demo.
- Make the proof-packet versus live-demo distinction explicit.

Allowed files or surfaces:

- `proof-packets/CONTACT_DRAFTS.md`
- `proof-packets/README.md`
- `proof-packets/browserbase-stagehand/README.md`
- `packages/integration/examples/browserbase-stagehand/README.md`
- `packages/integration/README.md`
- `LOOP.md`

Expected artifact or diff:

- Updated Browserbase draft with hosted demo placeholder or URL if deployed.
- Firecrawl channel and draft remain exact and human-gated.

Objective checks:

- `pnpm doc-sync`
- writing-rule scan for banned prose patterns.

Human gates:

- Posting Firecrawl comment.
- Opening Browserbase issue.
- Any X, Discord, email, or direct message.

### Packet 4: commit, push, and stop decision

Scope:

- Commit accepted local changes.
- Push the feature branch as a remote checkpoint.
- Stop before merge, deploy, or public posting unless the human gate clears.

Allowed files or surfaces:

- Git metadata for this worktree and branch.
- Agent Bridge progress note after a substantive push.

Expected artifact or diff:

- One commit for the live demo work.
- Pushed branch.

Objective checks:

- `git status --short`
- `git log -1 --oneline`
- `git ls-remote --heads origin codex/mcp-platform-proof-packets-20260623`

Human gates:

- Merge to `main`.
- Deploy hosted demo.
- Public comments, issues, emails, DMs, or posts.

## Feedback gate

Acceptance must come from checks, not maker output alone:

- Test/build/lint command for code.
- API or browser smoke for the demo surface.
- Generated artifact inspection for verifier output and proof links.
- Redaction scan for private Browserbase and Firecrawl material.
- Human judgment gate for deploy, merge, and all public contact.

## Stop rules

Stop when any condition fires:

- Accepted by verifier.
- Same failure twice.
- Missing permission.
- Unclear success criteria.
- Dirty or unsafe worktree.
- Human decision needed.
- Max iterations reached.

## Anti-gaming rules

- Do not loosen checks after seeing failures.
- Do not edit success criteria to fit a partial result.
- Do not skip required tests.
- Do not claim acceptance from maker output alone.
- Do not hide unresolved blockers.
- Do not post, comment, email, DM, merge, or deploy without a human gate.
