# Prior Art & Standards Map

Every specification, standard, and protocol that atrib builds on, extends, or hooks into. Organized by layer. Four relationship tags:

- **core:** atrib directly builds on this
- **extends:** atrib adds a layer on top of this
- **hooks:** atrib attaches to this at transaction time
- **context:** Relevant prior art or adjacent space

Draft 0.1, April 2026

## Layer 7: Agent Memory Layer (Composition-Adjacent)

The semantic-recall layer that runs inside an agent's conversation loop. atrib does not implement this layer; it runs underneath it. Listed here because the framing of "agent memory" overlaps semantically with atrib's "agents that reason from a past they can prove," and reviewers should see the distinction made explicit.

The triad operates on the conversational/semantic layer (what should the LLM read next turn?). atrib operates on the action layer (what action was emitted, by whom, in what causal context, and can a third party prove it?). Composition is clean: an agent can run any of the three for recall while running atrib middleware for signed action history. Cryptographic verifiability is the load-bearing difference, none of the triad signs records or anchors them in a transparency log.

## Layer 8: Agent Observability (Composition-Adjacent)

Agent observability platforms record what an agent does for operational debugging, cost tracking, evaluation, and root-cause analysis. atrib records the same activity for *verifiability*, every action becomes a signed Merkle-log entry any third party can audit without trusting the platform operator. The two layers compose rather than compete; atrib's primary observability integration is an `AtribSpanProcessor` that consumes OpenInference-shaped spans and emits signed atrib records on the same OTel pipeline.

## Layer 9: Agent Proof Logs (Close Alternatives)

Some newer tools do sign local agent activity. They are closer alternatives than generic observability tools because they share atrib's claim that agent actions need tamper-evident receipts.

## What No Existing Spec Provides

The gaps that atrib fills. None of the above specs address these:

| Gap                                         | Why it matters                                                                                                                                                                   |
| ------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Attribution record format**               | A signed, canonical data structure for recording that a specific creator's tool contributed to a specific session. OTel spans carry the data but are not signed by creators.     |
| **Value provenance graph**                  | A queryable graph connecting tool call events to transaction events via shared context, with typed edges (structural, co-occurrence, temporal). No existing spec defines this.   |
| **Attribution policy format**               | A machine-readable, negotiable document expressing how value should distribute across graph contributors. No existing spec defines weighting models for agent-mediated commerce. |
| **Settlement recommendation format**        | A verifiable output produced by applying a policy to a graph query ("Creator A gets 45%, Creator B gets 35%"), signed and auditable. Entirely novel.                            |
| **`atrib_context_id` in commerce metadata** | The convention for embedding an attribution context identifier in ACP/UCP checkout session metadata. Currently undefined in any commerce spec.                                   |
| **Cross-session recommendation token**      | A mechanism for attributing conversions that happen outside agent sessions (e.g., browser purchase after agent recommendation). Partially novel problem space.                   |
