# `dev.atrib/attribution`: Verifiable Attribution Records for MCP

| **Reserved `_meta` keys** | `dev.atrib/attribution` (requests and results) |
| **New methods / notifications / URI schemes** | None |
| **License** | Apache-2.0 |
| **Maintainer** | atrib (`hello@atrib.dev`), <https://github.com/creatornader/atrib> |

## 1. Overview

atrib makes the actions an AI agent takes provable: every tool call can become
an Ed25519-signed, chain-linked attribution record committed to a public
Merkle log, independently verifiable by anyone without trusting any operator.
The record format, log protocol, graph model, and verifier contract are
defined by the [atrib protocol specification](https://github.com/creatornader/atrib/blob/main/atrib-spec.md)
("the atrib spec" below). This document defines none of that; it defines only
how attribution context and attestation receipts travel over MCP as a
negotiated, first-class extension.

The extension declares three things and changes no signed byte of any atrib
record:

1. **Server-side signing capability.** A server advertises that tool calls
   here produce signed attribution records, with advisory metadata (event
   types signed, disclosure posture, expected creator key, log endpoints).
2. **Propagation carriage.** A vendor-prefixed `_meta` block on requests
   carrying exactly two fields in v0.1: the atrib propagation token and an
   explicit `context_id`, with a canonical two-ladder inbound resolution
   (Section 5).
3. **Attestation receipts.** When, and only when, the client declared the
   extension on that request, the server returns a receipt block in
   `result._meta` naming the record it just signed, optionally with the full
   signed record body for immediate signature re-verification.

The extension is a pure `_meta` + capability-map dialect. It defines no new
methods, no notifications, no `resultType` values, and no URI schemes. In the
stateless MCP model, `_meta` carries the protocol state that conformant
intermediaries are structurally obliged to forward; this extension is
**designed for `_meta` loss, not `_meta` theft**: every behavior degrades
silently to the pre-extension convention (Section 7).

### 1.1 Relationship to the unprefixed convention

atrib context has traveled over MCP since before this extension existed, as
an unnegotiated convention: the bare `_meta.atrib` key, a `tracestate`
`atrib=` entry, and an `X-Atrib-Chain` fallback, per atrib spec
[§1.5.4](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#154-mcp-transport-params_meta).
That convention remains fully supported and is the documented fallback at
every rung below the extension key. Implementations of this extension MUST
continue to read the legacy carriers and SHOULD continue to write them
alongside the prefixed block. The extension adds discovery, correct opt-in
gating, and a claimable namespaced surface; it is upside, not a dependency.

The key words MUST, MUST NOT, REQUIRED, SHALL, SHALL NOT, SHOULD, SHOULD NOT,

## 3. Extension Identifier and Versioning

The extension identifier is the string `dev.atrib/attribution`:
`{vendor-prefix}/{extension-name}` per SEP-2133, where the vendor prefix
`dev.atrib` is the reverse-DNS form of a domain the publisher controls, and
no DNS label of the prefix is `mcp` or `modelcontextprotocol` (those labels
are reserved for official extensions).

Versioning policy, per SEP-2133:

- The settings-object `version` field is `"0.1"` for this document.
- **Breaking changes require a new identifier** (e.g.
  `dev.atrib/attribution-v2`). The identifier published here is frozen: a
  peer that negotiates `dev.atrib/attribution` at any settings `version` MUST
  be able to interoperate with a v0.1 peer using only the behaviors in this
  document.
- **Non-breaking evolution uses settings fields.** New OPTIONAL settings
  fields, new OPTIONAL request-block fields, and new OPTIONAL receipt fields
  MAY be added in later settings versions. Receivers MUST ignore settings
  fields, request-block fields, and receipt fields they do not recognize
  (Sections 4, 5.1, 6.2).
- A peer declaring an unrecognized settings `version` under this identifier
  is treated as a valid declaration of this extension: the identifier, not
  the version string, is the compatibility unit.

Two fields are deliberately excluded from the v0.1 request block:
`session_token` (which already travels normatively in `baggage` under
`atrib-session`, atrib spec [§1.5.5](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#155-cross-trace-session-continuity))
and `provenance_token` (genesis-record-only host configuration, atrib spec
[§1.2.6](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#126-provenance_token)).
Either MAY be added only in a later settings version together with an
explicit conflict rule against its existing carrier and conformance vectors.

## 4. Capability Negotiation

### 4.1 Server capability declaration

A server implementing this extension advertises it in its capabilities under
the `extensions` map, in the `server/discover` response on stateless MCP
protocol versions, or in the `initialize` result on legacy protocol versions
(≤ 2025-11-25):

```json
{
  "capabilities": {
    "tools": {},
    "extensions": {
      "dev.atrib/attribution": {
        "version": "0.1",
        "signs": ["tool_call"],
        "receipts": ["token", "record"],
        "disclosure": { "args": "plain-sha256", "result": "omit" },
        "creator_key": "base64url-encoded 32-byte Ed25519 public key",
        "logs": ["https://log.atrib.dev/v1"],
        "directory": "https://directory.atrib.dev/v1"
      }
    }
  }
}
```

Server settings schema (v0.1):

| Field | Type | Required | Meaning |
|---|---|---|---|
| `version` | string | REQUIRED | Settings version, `"0.1"`. The only required field. |
| `signs` | string[] | OPTIONAL | atrib event-type short names this server signs (e.g. `"tool_call"`, `"transaction"`). |
| `receipts` | string[] | OPTIONAL | Receipt verbosities the server can produce: `"token"`, `"record"`. |
| `disclosure` | object | OPTIONAL | Advisory disclosure posture per atrib spec [§8.3](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#83-salted-commitment-posture) (e.g. `args`/`result` commitment forms). |
| `creator_key` | string | OPTIONAL | base64url-encoded 32-byte Ed25519 public key the server expects to sign with. |
| `logs` | string[] | OPTIONAL | Transparency-log base URLs records are submitted to. |
| `directory` | string | OPTIONAL | Identity-claim directory base URL for the creator key. |

Unknown settings fields MUST be ignored. A settings object missing `version`,
or whose `version` is not a non-empty string, is malformed; the peer MUST be
treated as not having declared the extension, and this MUST NOT produce a
protocol error.

### 4.2 Client capability declaration (per-request)

In the stateless MCP model, client capabilities travel per-request inside
`_meta` under the core-reserved key
`io.modelcontextprotocol/clientCapabilities`:

```json
"_meta": {
  "io.modelcontextprotocol/protocolVersion": "2026-07-28",
  "io.modelcontextprotocol/clientInfo": { "name": "my-agent", "version": "1.2.0" },
  "io.modelcontextprotocol/clientCapabilities": {
    "extensions": {
      "dev.atrib/attribution": { "version": "0.1", "accept": ["token", "record"] }
    }
  }
}
```

Client settings schema (v0.1):

| Field | Type | Required | Meaning |
|---|---|---|---|
| `version` | string | REQUIRED | Settings version, `"0.1"`. |
| `accept` | string[] | OPTIONAL | Receipt verbosity negotiation. `"token"` requests token-and-receipt-metadata; adding `"record"` additionally requests the full signed record body (Section 6.4). Defaults to `["token"]` when absent. Unknown values MUST be ignored; an `accept` array containing no recognized value is equivalent to `["token"]`. |

On legacy protocol versions where client capabilities travel in `initialize`,
the same settings object is declared under
`capabilities.extensions["dev.atrib/attribution"]` there, and it applies to
every request on that session.

### 4.3 Trust posture: declarations are advisory

Per SEP-2133's security requirements, extension-introduced data is untrusted.
Every server settings field beyond `version` is **advisory**: a hint for
pre-call decisions (a client or gateway pinning `creator_key`, a policy gate
checking a directory-published capability envelope per atrib spec
[§6.7](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#67-capability-declarations)).
Trust derives only from verifying signed records and Merkle-log inclusion
proofs, never from the capability map. A receipt whose `creator_key` differs
from a pinned declaration is a verifier-side signal, not a protocol error.

## 5. Request Carriage

The extension reserves the `_meta` key `dev.atrib/attribution` on requests.
The v0.1 block carries exactly two fields, both OPTIONAL; unknown fields MUST
be ignored (forward compatibility), and no other field is defined:

```json
"params": {
  "name": "search_web",
  "arguments": { "query": "..." },
  "_meta": {
    "dev.atrib/attribution": {
      "token": "base64url(record_hash).base64url(creator_key)  (≤87 chars)",
      "context_id": "32 lowercase hex characters"
    },
    "traceparent": "00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01",
    "tracestate": "atrib=<same token>",
    "baggage": "atrib-session=<session_token, when present>"
  }
}
```

### 5.1 Block schema

| Field | Type | Meaning |
|---|---|---|
| `token` | string | The atrib propagation token, unchanged from atrib spec [§1.5.2](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#152-http-transport-tracestate): `base64url(record_hash_bytes) + "." + base64url(creator_key_bytes)`, maximum 87 characters. Both decoded parts MUST be exactly 32 bytes; anything else is malformed. |
| `context_id` | string | The raw 32-character lowercase hexadecimal session anchor (atrib spec [§1.5.1](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#151-context_id-the-session-anchor)), carried explicitly per the stateless model. The MCP-transport analog of the `X-atrib-Context` HTTP header (atrib spec [§1.5.3.1](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#1531-context-id-header-x-atrib-context)). |

Clients implementing this extension SHOULD write the legacy carriers
(`tracestate` `atrib=` entry; optionally the bare `_meta.atrib` key and
`_meta["X-Atrib-Chain"]`) alongside the prefixed block, so that non-adopting
servers keep chaining.

### 5.2 Canonical inbound resolution, Ladder 1: propagation token

Two distinct values are resolved from an inbound request, so there are two
ladders. Servers implementing this extension MUST resolve the inbound
propagation token in this order, taking the first carrier that yields a
well-formed token:

```
1. _meta["dev.atrib/attribution"].token      (this extension)
2. _meta.atrib                               (legacy convention)
3. _meta.tracestate  "atrib=" entry          (atrib spec §1.5.2)
4. _meta["X-Atrib-Chain"]                    (atrib spec §1.5.3 fallback)
```

Rules:

- **Conflict.** When the extension key and a legacy carrier decode to
  different tokens, the extension key wins and the server SHOULD log a
  warning (atrib reference implementations prefix it `atrib:`). The request
  MUST NOT fail.
- **Lenient parse.** A malformed extension token (undecodable, wrong part
  lengths, missing separator) falls through to the next carrier. Malformation
  is never an error.
- **All carriers absent or malformed.** The inbound-token rung resolves to
  nothing and chain-root resolution continues down the multi-producer ladder
  of atrib spec [§1.2.3.1](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#1231-multi-producer-chain-composition)
  (within-process chain tail → env → mirror inheritance → synthetic genesis).
  This ladder refines only the "inbound propagation token" rung of that
  contract; the rest of the chain-root ladder is untouched.

### 5.3 Canonical inbound resolution, Ladder 2: context identity

Servers implementing this extension MUST resolve `context_id` in this order:

```
1. explicit context_id tool argument          (application intent)
2. _meta["dev.atrib/attribution"].context_id  (this extension)
3. _meta.traceparent trace-id                 (W3C Trace Context)
4. producer-local env/file discovery          (atrib D078/D083 registry)
5. undefined                                  (producer synthesizes; genesis)
```

Rules:

- **Argument beats transport.** An explicit `context_id` tool argument always
  wins: the extension block is transport metadata; an argument is application
  intent. On mismatch, the server uses the argument and SHOULD log a warning.
- **Strict format, lenient handling.** An extension-block `context_id` that
  is not exactly 32 lowercase hexadecimal characters is ignored and
  resolution falls through, never an error.
- **Extension beats traceparent.** When the extension block and `traceparent`
  disagree, the extension block wins with a warning. The trace-id rung
  remains for callers that carry no extension block, per the W3C traceparent
  validity rules (invalid or all-zero trace-ids are ignored).
- **Env/file discovery is last.** Producer-local resolution (environment
  variable and harness-registry file fallback) applies only when no
  per-request carrier resolved; its internal ordering is defined by the atrib
  decision log ([D078](https://github.com/creatornader/atrib/blob/main/DECISIONS.md#d078-mcp-servers-honor-atrib_context_id-env-as-context_id-default),
  [D083](https://github.com/creatornader/atrib/blob/main/DECISIONS.md#d083-harness-session-id-discovery-extends-d078-for-cognitive-primitive-mcp-servers)),
  not by this extension.
- **No effect on signed fields beyond `context_id`.** Unknown fields in the
  request block (including a `session_token` or `provenance_token` sent by a
  future or nonconforming peer) MUST be ignored and MUST NOT set any field
  of any signed record.

## 6. Result Carriage: Attestation Receipts

### 6.1 Gating

A server MUST emit the `dev.atrib/attribution` result block **only** when
the requesting client declared `dev.atrib/attribution` in that request's
`io.modelcontextprotocol/clientCapabilities` (or, on legacy protocol
versions, in the session's `initialize` capabilities). This is SEP-2133
opt-in gating.

- A missing, malformed, or non-object client declaration is treated as
  **undeclared**: the result block is omitted, no error is raised, and the
  tool result is otherwise byte-identical to pre-extension behavior.
- Gating governs **carriage only**. Whether a server signs its own actions is
  server-local policy; signing is not conditioned on the client's
  declaration, and signing does not require log submission.
- The legacy unprefixed result keys (`atrib`, `tracestate`,
  `X-Atrib-Chain`) predate this extension and continue to be written
  unconditionally, unchanged, regardless of negotiation state.

### 6.2 Receipt block

```json
"result": {
  "content": [ { "type": "text", "text": "..." } ],
  "_meta": {
    "dev.atrib/attribution": {
      "token": "base64url(record_hash).base64url(creator_key)",
      "receipt": {
        "record_hash": "sha256:<64 lowercase hex>",
        "creator_key": "base64url-encoded 32-byte Ed25519 public key",
        "context_id": "32 lowercase hex",
        "event_type": "tool_call",
        "chain_root": "sha256:<64 lowercase hex>",
        "log_submission": "queued"
      },
      "record": { "...full signed atrib record; only when accept includes \"record\"...": "" }
    },
    "atrib": "<same token; legacy key, always written>",
    "tracestate": "atrib=<same token>",
    "X-Atrib-Chain": "<same token>"
  }
}
```

Receipt schema (v0.1); unknown receipt fields MUST be ignored by consumers:

| Field | Type | Meaning |
|---|---|---|
| `token` | string | Propagation token of the record named by the receipt. MUST equal the token derived from that record (`base64url(sha256(JCS(record)))` + `.` + `base64url(creator_key_bytes)`). |
| `receipt.record_hash` | string | `"sha256:" + lowercase hex of sha256(JCS(signed record))`. MUST recompute from the record's canonical bytes when the record is available. |
| `receipt.creator_key` | string | The record's signer. MUST equal the record's `creator_key`. |
| `receipt.context_id` | string | The record's `context_id`. |
| `receipt.event_type` | string | atrib event-type short name (e.g. `"tool_call"`). |
| `receipt.chain_root` | string | The record's `chain_root`. |
| `receipt.log_submission` | string | Queue status: `"queued" \| "submitted" \| "disabled" \| "failed"` (Section 6.3). |
| `record` | object | OPTIONAL full signed atrib record (Section 6.4). |

A consumer that detects an internal inconsistency (token/`record_hash`/
`creator_key` disagreeing with the attached record, or with each other) MUST
treat the receipt as invalid and discard it. Receipt invalidity never
invalidates the tool result itself.

### 6.3 `log_submission` is a queue status, never an awaited proof

The receipt names a record the server has **already signed locally**.
`log_submission` reports the state of the server's non-blocking submission
queue at response time:

- `queued`: the record is enqueued for transparency-log submission.
- `submitted`: a submission round-trip has already completed.
- `disabled`: this server signs without log submission (a valid local-only
  posture).
- `failed`: submission attempts have failed; retries may continue.

Servers MUST NOT await log submission (or inclusion-proof retrieval) before
returning the tool result, per atrib spec
[§5.3.5](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#535-log-submission).
Inclusion proofs are fetched later, keyed by `record_hash`. A receipt is a
claim of local signing, not a claim of log inclusion; consumers requiring
inclusion verify against the log independently.

### 6.4 Full record body

When the client's `accept` includes `"record"`, the server SHOULD attach the
complete signed record under `record`. This is safe by construction: an atrib
record carries commitments (`args_hash`, `result_hash`, `content_id`), never
payloads, per atrib spec [§8.3](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#83-salted-commitment-posture).
Returning it enables immediate signature re-verification by the caller
without a mirror or archive round-trip. When `accept` is `["token"]` (or
defaulted), the server MUST omit `record`.

## 7. Degradation Behavior

Every extension behavior is subject to the atrib degradation contract
(atrib spec [§5.8](https://github.com/creatornader/atrib/blob/main/atrib-spec.md#58-degradation-contract)):
**extension processing must never affect the primary tool call or its
result.** Specifically:

- **`_meta` loss.** If an intermediary strips the extension block, or all of
  `_meta`, the request proceeds; resolution falls through the ladders and, at
  worst, a genesis record starts a new chain. Missing `_meta` MUST NOT block,
  delay, or alter the tool call.
- **Signing failure.** If record signing fails for any reason, the tool
  result is returned **without** the extension block and without error, byte-
  identical to passthrough.
- **Capability-read failure.** If reading or parsing capability declarations
  throws or yields garbage, the peer is treated as undeclared; the tool path
  sees no error.
- **Malformed inputs.** Malformed tokens, malformed `context_id` values, and
  unknown fields are ignored or fallen through, never surfaced as protocol
  errors.

Implementations SHOULD log degradations with an `atrib:`-prefixed warning on
their diagnostic channel; they MUST NOT write diagnostics into the JSON-RPC
message stream.

## 8. Interoperability Matrix

| Client | Server | Behavior |
|---|---|---|
| pre-extension | extension | Client never declares; server emits legacy result keys only; nothing breaks. |
| extension | pre-extension | Client writes both the prefixed block and legacy carriers; server reads the legacy ones (or none); the chain ladder falls through exactly as today. |
| extension | extension | Full negotiation: prefixed carriage inbound, gated receipts outbound, legacy keys still written for any downstream legacy hop. |
| pre-extension | pre-extension | The unprefixed convention, unchanged. |

## 9. Security Considerations

- **Capability declarations are untrusted input** (Section 4.3). They are
  advisory routing/pinning hints. Nothing in a capability map, request block,
  or receipt is trustworthy until a signature over canonical record bytes is
  verified and, where required, log inclusion is proven.
- **Receipts are claims, not proofs.** A malicious server can emit a receipt
  for a record it never signed or never submitted. Consumers verify: (1) the
  receipt's internal consistency, (2) the record's Ed25519 signature over its
  JCS-canonical bytes, (3) log inclusion via an independently fetched proof
  bundle. Steps 2-3 need no cooperation from the server.
- **Key substitution.** A server may sign with a different key than its
  advertised `creator_key`. Pinning plus directory-published identity claims
  and capability envelopes make this detectable; it is a trust signal, not a
  transport error.
- **Privacy.** The request block carries only a hash-derived token and a
  session identifier; the receipt and record carry commitments, not payloads.
  Intermediaries that forward `_meta` learn linkage (that two calls share a
  session), which is already true of `traceparent`/`tracestate`. Operators
  for whom session linkage is sensitive should scope `context_id` lifetimes
  accordingly.
- **Injection.** Servers MUST treat request-block fields as data, never as
  code or as record fields other than the two defined mappings (`token` →
  chain-root resolution; `context_id` → `context_id`). Unknown fields MUST
  NOT flow into signed records (Section 5.3).
- **Denial of service.** Ladder resolution is constant-work per carrier;
  malformed carriers fall through in O(1). Receipt production is bounded by
  the record the server already produced.

## 10. Conformance

An implementation conforms to `dev.atrib/attribution` v0.1 if it satisfies
every MUST in Sections 3-7 for the role(s) it implements (client, server, or
both). The machine-checkable corpus at
[`spec/conformance/mcp-extension/`](https://github.com/creatornader/atrib/tree/main/spec/conformance/mcp-extension)
pins six case families: capability declarations, negotiation gating, token
precedence (Ladder 1), context-identity precedence (Ladder 2), receipt
integrity against real signed records, and degradation. Implementations
SHOULD validate against the corpus; the vectors are the shared contract
between independent implementations in any language, with no atrib package
required.
